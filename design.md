# 家庭收纳备忘网页方案

## 一、核心目标
- 搭建一个多页面网站，帮助全家人记录和管理家中物品的存放位置。
- 按 **住所 → 房间 → 收纳位 → 物品** 三级结构组织。
- 支持多人协作、家庭分组、权限隔离、拍照上传。
- 完全使用免费服务（Cloudflare 全家桶），零运营成本。

---

## 二、技术选型与免费配额
所有服务均部署在 Cloudflare 生态内，免费额度足以满足家庭日常使用。

| 服务 | 用途 | 免费额度 | 注意事项 |
|------|------|----------|----------|
| Cloudflare Pages | 托管前端多页静态资源 + 后端 API（Pages Functions） | 无限带宽，500 次构建/月 | — |
| Pages Functions | 提供同域 API（无跨域） | 10 万请求/天 | 免费计划单次请求 CPU 时间仅 **10ms**，图片压缩等耗时操作需在客户端完成 |
| D1（Cloudflare 的关系型数据库） | 存储用户、家庭组、住所、房间、收纳位、物品等数据 | 5 GB 存储，500 万行读取/月 | 分布式数据库冷查询有延迟，热点数据前端做适当缓存 |
| R2（Cloudflare 的对象存储） | 存放物品照片 | 10 GB 存储，100 万次写，1000 万次读/月 | 写次数充足，但上传失败重试需做退避策略 |
| 自建认证（JWT） | 用户注册/登录，签发 JWT token | 无额外服务需要 | 详见第三节安全设计 |

## 三、用户与家庭分组方案

### 1. 用户认证
- 每个家庭成员拥有独立账号（邮箱 + 密码 + 邀请码注册/登录）。
- **注册即入组**：注册时强制要求邀请码（创建家庭除外），注册成功后自动成为对应家庭组的成员，不存在"先注册后加入"的分离流程。
- **密码要求**：最少 8 位，必须包含字母和数字。
- **JWT 存储方式**：后端签发 JWT 后，以 `httpOnly` + `Secure` + `SameSite=Strict` 的 Cookie 形式返回，前端不直接接触 token，防止 XSS 窃取。
- 所有页面在加载时自动校验 JWT 有效性，失效则跳转到登录页。
- **Token 刷新**：JWT 有效期设为 15 分钟，同时签发有效期 7 天的 Refresh Token（同样 httpOnly Cookie），前端无感知自动刷新。
- **CSRF 防护**：使用 `SameSite=Strict` Cookie + 自定义请求头（如 `X-Requested-With: XMLHttpRequest`）双重校验。
- **速率限制**：登录/注册接口限制为同一 IP 每分钟 10 次请求（注册接口加 Cloudflare Turnstile 人机验证），防止暴力破解和恶意注册。
- **防恶意注册**：没有有效邀请码无法完成注册。公网暴露的注册页仅对持有邀请码的家庭成员开放，陌生人即使访问也无法创建账号。

### 2. 注册流程（两条路径）

注册页提供两个入口，用户根据自身情况选择：

**路径 A：创建家庭（首个用户）**
1. 用户点击"创建新家庭"，进入创建家庭注册表单。
2. 填写邮箱、密码、昵称、家庭名称。
3. 提交后，后端同时创建用户和家庭组，用户自动成为 `admin`。
4. 系统生成唯一邀请码（8 位字母数字），管理员可分享给家人。

**路径 B：加入家庭（家庭成员）**
1. 用户从管理员处获得邀请码。
2. 用户点击"加入家庭"，进入加入家庭注册表单。
3. 填写邮箱、密码、昵称，并输入邀请码。
4. 后端校验邀请码（有效、未过期、未达最大使用次数），通过后创建用户并自动加入对应家庭组，角色为 `member`。

```
注册页
├── "创建新家庭" → 无邀请码，填写家庭名称，成为 admin
└── "加入家庭"   → 需邀请码，自动加入，成为 member
```

### 3. 家庭组创建与管理
- 管理员可在成员管理页查看/复制邀请码、重置邀请码、设置邀请码最大使用次数（默认 10 次），也可将成员移出家庭组。
- 邀请码有效期默认 7 天，过期后管理员需手动重置。
- 一个用户可以同时属于多个家庭组（分别经由不同邀请码注册加入），登录后在家庭组列表页切换。

### 4. 数据隔离与权限
- 所有核心数据（住所、房间、收纳位、物品）均与一个 `family_id` 绑定。
- 每个 API 请求都会验证：当前用户是否属于该 `family_id` 的成员。
- **管理员** 可以增删改家庭组内的所有数据，管理成员。
- **普通成员** 可以增删改该组内的所有物品数据（家庭成员之间数据完全共享），但无法修改家庭组设置、无法移出其他成员。
- 不同家庭组之间的数据彻底隔离，互不可见。

## 五、三级结构详情

### 1. 住所
- 属性：名称（完全自定义，如"北京的家"、"父母家"）。
- 操作：添加、删除（软删除，删除住所会级联软删除其下所有房间、收纳位和物品，操作前二次确认）。

### 2. 房间
- 属性：名称。
- 预设名称列表：客厅、主卧、次卧、厨房、卫生间、阳台、储物间、书房。
- 添加时提供 `<datalist>` 建议预设值，但允许手动输入完全自定义的名称。
- 每个房间必须属于一个住所。

### 3. 收纳位
- 属性：名称。
- 预设名称列表：衣柜、床头柜、电视柜、书桌抽屉、收纳箱、吊柜、鞋柜、橱柜。
- 同样支持预设 + 自定义输入。
- 每个收纳位必须属于一个房间。

### 4. 物品
- 属性：名称（必填）、分类、数量、备注、创建时间、修改时间。
- 每个物品必须属于一个收纳位（保证位置精确）。
- **物品移动**：支持将物品从一个收纳位移到另一个收纳位（含完整的位置选择器），保留所有照片和历史记录。
- 支持按名称模糊搜索，按分类筛选。

### 5. 物品分类
- 系统预设分类：证件、数码、工具、药品、衣物、书籍、日用、食品、其他。
- 家庭管理员可自定义分类名称，分类数据存储在独立的 `categories` 表中，避免同义词混乱。

## 六、悬浮添加按钮设计（全局）
每个页面（住所、房间、收纳位、物品列表）右下角固定一个圆形蓝色 "+" 悬浮按钮。

- 点击后弹出两个横向选项：
  1. **"添加当前级"**
     - 在住所页 → 显示"添加住所"
     - 在房间页 → 显示"添加房间"
     - 在收纳位页 → 显示"添加收纳位"
     - 点击后弹出内联表单或模态框，填写名称后保存，列表即时刷新。
  2. **"直接添加物品"**
     - 点击后从屏幕底部弹出一个半屏面板（底部弹窗），用于快速记录物品。
     - 弹窗内容：
       * 物品名称输入框
       * 分类下拉（预设 + 自定义）
       * 数量输入框
       * 备注输入框
       * 拍照按钮（调用手机相机或相册）
       * 位置选择器（住所 → 房间 → 收纳位三级联动下拉，动态加载当前家庭组的数据）
       * 保存按钮

### 智能默认位置
- 在住所首页点击"添加物品"，位置选择器无默认值，需手动选择住所。
- 在房间页（已知住所）点击，住所自动选中该房间所属的住所，房间自动选中当前房间，收纳位需手动选。
- 在收纳位页（已知房间和收纳位）点击，住所、房间、收纳位均自动选中，可直接修改。
- 若在物品列表页点击，同样自动选中当前收纳位，方便为该位置继续添加物品。

### 三级联动性能优化
- 前端在页面加载时一次性拉取当前家庭组的所有住所→房间→收纳位树形数据并缓存。
- 选择器直接使用本地缓存数据做联动，避免每次切换都发起 API 请求。

## 七、拍照上传方案（完全免费）
1. 前端使用 `<input type="file" capture="environment" accept="image/*">` 拍照或选图。
2. 图片在客户端压缩（Canvas API，压缩至 800px 宽，JPEG 质量 0.7），降低上传量和存储占用。
3. 以 `multipart/form-data` 格式 POST 到 `/api/v1/upload`。
4. Pages Functions 接收文件，生成唯一文件名（`{familyId}/{timestamp}_{random}.jpg`），并上传至 R2 存储桶。
5. 上传成功后，后端返回文件的 R2 公开访问 URL。
6. 前端将图片 URL 保存到物品的 `item_photos` 关联表中。
7. 物品详情中，以缩略图网格展示照片，点击可放大查看。

**注意**：图片压缩须在客户端完成，Pages Functions 免费计划 CPU 时间有限（10ms），不适合做服务端图片处理。

## 八、数据库设计概要

### 1. 表结构

- `users`：id, email, password_hash, name, created_at, updated_at, deleted_at
- `families`：id, name, created_by, invite_code, invite_code_expires_at, created_at, updated_at, deleted_at
- `family_members`：id, user_id, family_id, role (admin/member), joined_at
- `categories`：id, family_id, name, sort_order, created_at
- `houses`：id, family_id, name, created_at, updated_at, deleted_at
- `rooms`：id, house_id, name, created_at, updated_at, deleted_at
- `storage_spots`：id, room_id, name, created_at, updated_at, deleted_at
- `items`：id, storage_spot_id, category_id, name, quantity, note, created_at, updated_at, deleted_at
- `item_photos`：id, item_id, url, sort_order, created_at
- `activity_logs`：id, family_id, user_id, action (create/update/move/delete/restore), entity_type, entity_id, entity_name, details (JSON), created_at

### 2. 关键设计说明

**软删除**：
所有核心实体表（users, families, houses, rooms, storage_spots, items）均包含 `deleted_at` 字段。删除操作仅设置时间戳，数据不物理删除。回收站页面展示已删除数据，支持恢复（清空 `deleted_at`）或彻底删除。

**照片拆分**：
原方案中 `items.photos` 作为 JSON 数组存储，不利于独立管理和查询。改为独立的 `item_photos` 表，支持排序和独立增删。删除物品时级联软删除其所有照片。

**操作日志**：
`activity_logs` 表记录所有增删改移操作，包含操作人、操作类型、操作对象、详情快照。物品详情页底部展示该物品的操作时间线。

**分类独立**：
`categories` 表存储家庭组级别的自定义分类，管理员可增删改。物品通过 `category_id` 外键关联，解决自由文本导致的同义词问题。

### 3. 数据库级约束

- 所有外键使用 `ON DELETE CASCADE`，数据库层面保证级联一致性。
- 需要建索引的字段：`families.invite_code`（唯一索引）、`users.email`（唯一索引）、所有 `family_id`、`house_id`、`room_id`、`storage_spot_id`、`deleted_at` 等查询和关联字段。
- `items.name` 建立全文索引或普通索引，支持全局模糊搜索。

### 4. 数据隔离链路
所有涉及家庭数据隔离的表，均通过 `family_id` 或其上级外键（如住所带 `family_id`）实现。查询时的隔离路径为：

```
物品 → storage_spot_id → room_id → house_id → family_id
```

## 十、前端架构规划

### 1. 共享模块
所有 HTML 页面通过 `<script src="js/xxx.js">` 引入共享模块，避免重复代码：

| 模块文件 | 职责 |
|----------|------|
| `js/api.js` | 封装 fetch 请求，自动附带认证 Cookie 和 CSRF 头，统一错误处理 |
| `js/auth.js` | 认证状态检查、自动跳转登录、Token 刷新 |
| `js/ui.js` | 面包屑生成、底部导航栏、悬浮添加按钮、模态框、Toast 通知 |
| `js/components.js` | 卡片组件、空状态、加载骨架屏、分页器、确认弹窗 |

### 2. 全局状态管理
- 当前登录用户信息、当前选中的家庭组 ID 存储在模块级变量中。
- 三级联动数据（住所→房间→收纳位树）在进入家庭组时一次性拉取并缓存。

### 3. UI 状态覆盖
每个数据展示区域必须覆盖以下 4 种状态：

| 状态 | 表现 |
|------|------|
| 加载中 | 骨架屏（卡片灰色占位）或列表级 spinner |
| 空数据 | 居中插图 + 引导文案（如"还没有收纳位，点击右下角 + 添加"） |
| 正常 | 数据卡片列表 |
| 错误 | 错误提示 + 重试按钮（网络错误、权限不足等） |

## 十一、界面与交互设计要点
- **移动端优先**：大按钮、大字体，适合手机在找东西时单手操作。
- **卡片式列表**：每个实体用卡片展示，一眼看到名称和统计信息（如"5 件物品"）。
- **面包屑导航**：顶部显示：家庭组名 > 住所名 > 房间名 > 收纳位名，任何一级可点击跳转。
- **快速搜索与筛选**：在物品列表页顶部提供搜索框和分类筛选下拉，支持即时过滤。
- **全局搜索入口**：底部导航栏常驻搜索图标，支持跨所有收纳位搜索物品名称。
- **删除安全**：任何删除操作都需要弹窗确认。删除后进入回收站，30 天后自动清理或手动彻底删除。
- **乐观更新**：物品列表的增删改操作先更新 UI，再发 API，失败时回滚并提示。

## 十二、部署流程概要
1. 注册 Cloudflare 账号。
2. 使用 Wrangler CLI 创建 Pages 项目，并绑定 D1 数据库和 R2 存储桶。
3. 在 D1 中执行建表 SQL（含所有索引和外键约束）。
4. 编写前端 HTML/CSS/JS 和后端 Functions 代码。
5. 将代码仓库连接至 Cloudflare Pages，设置构建输出目录为 `/`。
6. 配置环境变量（`JWT_SECRET`、`JWT_REFRESH_SECRET`、`R2_BUCKET_NAME` 等）。
7. 部署，获得 `xxx.pages.dev` 域名，也可绑定自定义域名。
